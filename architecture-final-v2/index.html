The shape.

01 / SHAPE

Atlas OS is not a single product. It is a coordinated set of brand-scoped operators, each bound to a brand, all sharing one semantic memory mesh and one operator identity layer. The pattern is intentional. One brand cannot leak into another. One operator cannot lose context across sessions.

Three layers, top to bottom:

  • Identity layer. Each bot carries SOUL, IDENTITY, USER, and MEMORY files that define voice, scope, allowlist, and brand-specific facts. These are the prompts the runtime injects on every turn.
  • Runtime layer. Hermes Agent v0.11 on most boxes. OpenClaw still active on five brand boxes pending migration. Tommy uses a custom FastAPI bridge.
  • Memory layer. Postgres 16 with pgvector on Atlas. Six schemas, one per agent class. 768 dimension embeddings. Cross-schema reads gated by Postgres roles.

The memory mesh.

04 / MEMORY

One Postgres 16 instance on Atlas, port 5432, localhost-bound. pgvector extension. Six schemas. Every bot writes its own. Cross-bot reads gated by Postgres role grants.

SchemaOwnerPurpose
atlas_memoryatlas_adminAtlas operator working memory and embeddings.

Embedding model is OpenAI text-embedding-3-small, 768 dimensions. Same model across the mesh, intentionally. Lets any bot query any indexed schema and get useful similarity scores.

The runtime stack.

05 / RUNTIME

Hermes Agent v0.11 (Atlas, Arlo, Zen)

Modern runtime. One systemd unit per bot. Shared venv. Per-bot HERMES_HOME directory holds SOUL, MEMORY, skills, workspace. Skills load on demand. Tool use is structured and logged.

/home/ubuntu/.hermes/hermes-agent/    # shared install
/home/ubuntu/.hermes-disruptive/      # Atlas SOUL + memory
/home/ubuntu/.hermes-arlo/            # Arlo SOUL + memory

systemctl status hermes-disruptive
systemctl status hermes-arlo

OpenClaw (Charles, Otto, Revive, 3D, CBum, Nora)

Legacy runtime. Telegram polling loop, agent dispatch, memory in flat files plus pgvector. Stable, but skill ecosystem and tool framework lag Hermes. Migration is ordered, not urgent.

Hermes voice and scope discipline

Both Hermes bots run with hard rules baked into SOUL. No em dashes. Brand isolation. No cross-injection of secrets. Pre-flight gate before infrastructure mutations. Make-by-Kaeden attribution on all artifacts.

Access and identity.

06 / ACCESS

AWS

  • Disruptive Brands AWS account: 225726002432. Region us-east-1.
  • Atlas runs with EC2 instance role DisruptiveAIBotRole. Native role on the box, not assumed.
  • Brand boxes carry brand-scoped roles. Read-only for Atlas where possible, write where needed.
  • Zen Control reaches the Disruptive fleet via cross-account assume role from a separate management account.

Secrets

  • AWS Secrets Manager is the single source of truth. JSON-wrapped values, accessed with jq -r .value.
  • Telegram tokens, Anthropic keys, Monday API token, RAW backend creds, all live in Secrets Manager.
  • Each bot pulls only its own secrets. No cross-brand secret access.

Telegram allowlists

  • Disruptive bot: Miguel, Dom, Matty, Kaeden. Group chats: explicit allowlist.
  • Arlo: Kaeden plus authorized Revive content team.
  • Brand bots: brand-team-only.

SSH and SSM

  • SSH is disabled or behind tight SG rules on production boxes.
  • SSM Session Manager is the operator path. aws ssm start-session.
  • From Atlas, helper script atlas-ssm.py wraps fleet-wide SSM with shortcuts (atlas-ssm raw "uptime").

Architectural exceptions.

08 / EXCEPTIONS

TOMMY IS NOT A NORMAL BOT

Tommy on RAW Athletic Club does not run an OpenClaw or Hermes Telegram polling loop. The RAW FastAPI backend owns Telegram webhook ingress. The backend calls openclaw agent --local through tommy_bridge.py for the LLM brain. Do not enable a polling loop on the Tommy host. Read knowledge/playbooks/tommy-architecture.md before touching anything on RAW Athletic.

ATLAS HOLDS PRIVILEGED ACCESS

Atlas can SSM into every Disruptive box. Treat the Atlas SOUL and identity files as a control surface. Changes to its allowlist, scope, or skills should go through the same review as a production deploy.

ZEN IS NOT A DISRUPTIVE BOT

Zen lives on a separate management account. It is Kaeden's operator, scoped to operate Disruptive infrastructure on his behalf. It is not bound to a brand. Do not put Zen in brand allowlists.

Tracked on the AI Board in Monday. Sequenced by impact and dependencies. Items below reflect open scope as of April 28.

  • Amazon integration for all stores. Scale the TACOS pattern across the portfolio.
  • Revive SEO and AEO program. Sustained content cadence with citation discipline.

90 days. Atlas OS as a product.

  • Disruptive Command Center. Unified ops UI on top of the existing infrastructure.
  • Atlas OS SaaS scope decision. Define the MVP, declare what is and is not in the box.
  • Productize the brand-box pattern. Spin up a new brand bot in a day, not a week.

Open questions.

10 / OPEN

  • Atlas OS SaaS. Are we productizing this externally, or keeping it internal as a competitive moat? Decision unblocks Command Center scope and integration roadmap.
  • Cross-brand semantic search. Hive mind schema exists, retrieval surface does not. Worth building, but only after migration finishes.
  • Dashboard surface. Where does ops data live for the team day to day. Sheets work today. Command Center is the next leap.
  • Contractor onboarding velocity. Gustavo onboarded in days. Pattern is reusable, but needs codifying as a playbook.
THE BET

Operator-led, AI-native, brand-isolated. Different from a single chatbot, different from a generic agent platform. The architecture is the moat. Every bot the team trusts is one fewer thing the team has to remember.
