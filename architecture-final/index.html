Design principle

Isolated lanes, shared backbone

The architecture is not one giant bot on one giant box. It is a shared backbone with isolated brand lanes underneath it. That keeps control centralized while keeping brand execution separate.

OpenClaw deployment model

How OpenClaw runs across the stack

OpenClaw is the runtime layer that gives each lane its agent behavior, memory handling, tools, and orchestration logic. It is the software layer sitting on top of the server infrastructure.

  • per-server agent runtime
  • per-brand workspaces
  • session and memory handling
  • tool execution and automation
  • message orchestration
  • operator behavior per lane

Memory model: Atlas hosts the shared PostgreSQL + pgvector memory backbone, and the bots connect into it remotely with isolated roles while also keeping local workspace context such as SOUL, MEMORY, and knowledge files.

Operational use

How Miguel uses Atlas OS

Atlas OS is not meant to be used like a traditional server farm. The intended usage model is operational, not infrastructure-first. Miguel uses the system through the brand bots, workflows, approvals, and dashboards that sit on top of the server stack.

What it should do for Miguel

  • run brand-specific bots in isolated lanes
  • handle brand workflows and operational tasks
  • surface approvals, alerts, and follow-up actions
  • give visibility into what is running underneath

What it is not for

  • casual direct server usage every day
  • cross-brand access without boundaries
  • shared credentials floating across the stack
  • one monolithic bot handling everything

Best mental model: Miguel should use the system through the operating layer, while the server and OpenClaw architecture stays controlled underneath it.

Admin access

How to log in and find the fleet

For infrastructure-level access, the entry point is Miguel’s existing AWS account access in us-east-1. Once logged into AWS, the server fleet is visible under EC2. All 7 instances are online in Systems Manager.

AWS Console
→ us-east-1
→ EC2
→ Instances
→ Select server
→ Use EC2 details for visibility
→ Use Systems Manager for admin shell access when needed

Important: treat AWS console + Systems Manager as the standard admin access path. Direct SSH is not universal across the fleet and should be treated as exception-only.

Security model

How AWS security works here

  • the environment lives inside Miguel’s AWS account
  • each brand lane is isolated at the server level
  • IAM roles are scoped per server or lane
  • secrets are intended to come from AWS Secrets Manager
  • the stack is designed around controlled administrative access
OpenClaw security

How runtime security works

  • each bot has its own workspace context
  • memory is shared at the infrastructure layer but isolated by role
  • local SOUL, MEMORY, and knowledge files stay per bot
  • brand lanes are meant to avoid cross-brand leakage
  • OpenClaw is the runtime, not the owner of the infrastructure
Emergency control

Backup and kill switch

Server-level stop per lane after export backup is the cleanest emergency control model for the stack.

Backup evidence: the bucket zero-client-miguel-225726002432 exists, and migration artifacts are already present under migration-v2/.
